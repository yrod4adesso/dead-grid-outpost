# Follow-Up Consequences Spec

## Goal

Give day events and outpost tasks short-term memory so small decisions create visible pressure or payoff on the following day instead of disappearing immediately after resolution.

This slice should deepen the current loop without requiring a broad content expansion. The point is not to add many new blueprints, but to make the existing event/task systems leave a track in the run state.

## Why This Slice Now

The prototype now already has:

- explicit run lifecycle
- persistence and regression coverage
- stronger threat escalation
- stronger survivor wear and readiness pressure

That means the next missing depth layer is decision memory:

- events currently resolve and vanish
- tasks currently resolve and rotate without much downstream effect
- the player still feels many day decisions as local rather than cumulative

This slice gives the run a stronger sense of consequence with relatively low content cost.

## Scope

This slice includes:

1. Follow-up state generated by selected day-event outcomes
2. Follow-up state generated by selected task outcomes
3. Next-day payoff or penalty hooks into at least two systems
4. UI surfacing so the player can see pending consequences
5. E2E coverage for persistence and visibility

This slice does not include:

- large new event chains
- major route-family expansion
- day modifier system
- full narrative branching
- additional recruit archetypes

## User Outcomes

After this slice:

- event choices can create tomorrow pressure or opportunity
- some tasks can reduce or worsen later threat
- the player can see that the run remembered a prior decision
- recent actions shape planning beyond the current click

## Product Behavior

### 1. Pending Consequences Layer

The game should support lightweight scheduled consequences that survive save/load and resolve on a later day.

Recommended structure:

- consequence created today
- consequence marked as pending
- consequence applied at next-day transition or next outpost phase
- consequence logged visibly when it lands

This should be modeled as explicit state, not only inferred from text.

### 2. Event Follow-Ups

Selected event outcomes should be able to schedule next-day effects.

Recommended starter patterns:

1. Risky event choice
   - grants better immediate reward
   - schedules next-day strain such as:
     - threat bump
     - weaker route intel
     - small supply loss

2. Safer event choice
   - smaller immediate reward
   - schedules smaller or positive follow-up such as:
     - lower threat carry
     - safer route condition
     - support resource boost

The first slice only needs one or two concrete examples, but the system should generalize.

### 3. Task Follow-Ups

Specific tasks should influence the next cycle rather than only adding same-day resources.

Recommended starter patterns:

1. Defensive prep task
   - reduces later night pressure
   - or improves defense readiness summary

2. Logistics / ration task
   - reduces next-day resource strain
   - or prevents a negative carry-over

3. Ignored opportunity
   - if a task rotates out unresolved, it may convert into a small penalty

The first slice can begin with resolved-task follow-up only if unresolved-task penalties feel too wide for one pass.

### 4. Visibility

Pending consequences must be visible before they resolve.

Recommended surfacing:

- a `Pending consequences` panel or commander note on the dashboard
- a concise label on affected event/task resolution text
- activity log entries when a scheduled effect is created
- activity log entries when the consequence actually lands

The player should never feel like the game changed something “off-screen” without warning.

## State / Architecture Changes

### New State Layer

Add a small explicit state container for follow-up consequences.

Suggested shape:

- `pendingConsequences: PendingConsequence[]`

Each entry should minimally contain:

- stable `id`
- source type: `event` or `task`
- source id / title
- trigger timing, likely `next_day`
- effect type
- effect payload
- player-facing label

The exact type names can change, but the data model should remain explicit and serializable.

### Trigger Model

Recommended initial rule:

- consequence is created on resolution
- consequence resolves automatically when the next day begins after a successful night cycle

This keeps the first slice simple and deterministic.

### Effect Hooks

Pending consequences should be able to affect at least these systems:

1. threat
2. resources
3. route pressure / route copy
4. defense briefing or reward context

Only a subset needs to ship in the first implementation, but the model should allow extension.

## UI Changes

Recommended UI additions:

1. Commander / dashboard
   - `Pending consequences` area with 1-3 concise rows

2. Event resolution feedback
   - tell the player when a choice created tomorrow pressure or opportunity

3. Task resolution feedback
   - tell the player what later effect was prepared

4. Activity log
   - creation log
   - resolution log

The UI should explain both:

- what is pending now
- what landed when the next day started

## Acceptance Criteria

### State

- At least one event choice can create a pending next-day consequence
- At least one task can create a pending next-day consequence
- Pending consequences persist across reload
- Pending consequences resolve deterministically on the intended next cycle

### Gameplay

- The player can feel at least one negative follow-up and one positive follow-up
- At least two gameplay systems are affected across the implemented examples
- The consequence changes more than flavor text

### UX

- Pending consequences are visible before they resolve
- Resolved consequences are logged clearly
- The player can understand which earlier decision caused the new effect

### Regression Safety

- Existing run lifecycle flows still work
- Threat / survivor pressure behavior still works
- Current persistence tests still pass

## Test Plan

Add or extend Playwright coverage for:

1. event choice creates pending consequence and survives reload
2. task completion creates pending consequence and survives reload
3. next successful day transition resolves the pending consequence
4. UI surfaces both pending and resolved consequence states

Suggested test style:

- use deterministic localStorage shaping when needed
- prefer validating both state-driven UI and resulting log text

## Implementation Tasks

### Task 1: Pending Consequence State Model

- define serializable consequence type
- add it to `DeadGridState`
- normalize and hydrate it safely

### Task 2: Event Follow-Up Hook

- add one risky and one safer next-day consequence path to an existing event blueprint
- log consequence creation clearly

### Task 3: Task Follow-Up Hook

- add at least one task that schedules a useful next-day effect
- optionally add one simple neglect / rotation penalty if scope stays small

### Task 4: Resolution Engine

- apply pending consequences at the next-day transition
- remove resolved entries cleanly
- log the result in activity feed

### Task 5: UI Surfacing

- show pending consequences in the dashboard
- tighten event/task feedback copy

### Task 6: E2E Coverage

- add deterministic tests for scheduling, persistence, and landing of consequences

## Risks

- Too many effect types too early will make the system noisy
- If pending consequences are hidden, the slice will feel unfair
- If penalties are stronger than rewards, the system may feel punishing instead of memorable

## Recommended Delivery Order

1. Pending consequence state model
2. Event consequence creation
3. Task consequence creation
4. Next-day resolution engine
5. Dashboard visibility
6. E2E coverage

## Definition Of Done

This slice is done when:

- events and tasks can create explicit next-day consequences
- those consequences survive reload and resolve cleanly
- the dashboard makes pending effects legible
- the activity log explains both creation and resolution
- regression coverage still passes
